# 控制平面 HA — Phase 2 實作設計（狀態外移 + 選主）

> 上層設計見 [ha-design_zh-CN.md](ha-design_zh-CN.md)。本文是 **Phase 2 的可執行工作分解**：
> 具體改哪些檔、怎麼做、怎麼測、風險與驗收。目標是讓 backend 能跑 **多副本、無單點故障**。
> Phase 1（優雅排空）已完成。

## 0. 總原則（再強調）

- **零破壞單機**：不設 `LLMOPS_DB_URL` → 一切照舊走 SQLite + 本機 overlay 檔，零設定可跑。
- **可分批上線**：拆成 2a–2e 五個子階段，**每個都能獨立合併、獨立有價值**。最關鍵也最先做的是
  2a（store 抽象化）—— 它本身就讓「換 Postgres 備份/查詢」可用，且是其餘子階段的地基。
- **冪等 + best-effort**：所有寫入可重入；HA 機制失敗時退化為「單副本正常運作」，不得讓狀態機卡死。

## 1. 現況（Phase 2 要動的地方）

| 元件 | 現況 | 障礙 |
|---|---|---|
| store | `packages/llmops-store/llmops_store.py` 單一 **aiosqlite** 類別；backend([app/core/store.py](../apps/backend/app/core/store.py))與 router([store.py](../apps/router-server/src/llm_router/store.py))各自 sys.path 重新匯出**同一個類別** | SQL 充滿 SQLite 方言：`?` 佔位、`AUTOINCREMENT`、`lastrowid`、`INSERT OR IGNORE`/`ON CONFLICT`、`PRAGMA`、`PRAGMA table_info` migration |
| overlay | [overlay.py](../apps/backend/app/services/overlay.py) 讀寫單一 JSON 檔 `data/dynamic_models.json` | 檔案無法跨機器安全共用；多副本各看各的 |
| desired 狀態 | 只在 registry 記憶體（`inst.desired`） | 重啟即失憶；副本無法接管 |
| 背景迴圈 | [main.py](../apps/backend/app/main.py) lifespan 起 6 個 in-process 單例（reconcile / autoscaler / gpu / load / 2×prune） | 多副本 = 多個 reconciler/autoscaler 打架 |
| router 狀態 | `metrics_cache`/`backend_inflight`/`rr_counters` 在 `app.state`；`workers=1` | 多副本 inflight 與 RR 游標不一致 |

## 2a. Store 抽象化 + Postgres 後端（地基，零行為改變）  ✅ 已完成

**目標**：`LLMOPS_DB_URL=postgres://…` 走 Postgres，否則沿用 SQLite；對外介面與行為完全不變。

> 進度：**已完成**。`packages/llmops-store/_driver.py`（`SqliteDriver` / `PgDriver`(asyncpg) +
> `?`→`$n`、`AUTOINCREMENT`→`IDENTITY`、`REAL`→`DOUBLE PRECISION`、`RETURNING id`、`ensure_column`）；
> `LLMOpsStore` 改走 driver（6 個 insert 用 `insert()`、migration 用 `ensure_column()`）。
> 測試：store 全套**參數化跑 SQLite + Postgres 兩種後端**（`LLMOPS_TEST_DB_URL` 開,缺則 skip）+
> driver 方言純函式單元測試。SQLite 行為零退化（backend 334 / router 109 / store 28，PG 變體再 +20）。

### 做法
1. **方言 driver 薄層**（新 `packages/llmops-store/_driver.py`）：
   - 抽象 `Driver`：`execute(sql, params)`、`fetchone`、`fetchall`、`insert(sql, params) -> id`、
     `executescript`、`transaction()`。
   - `SqliteDriver`（aiosqlite，保留 WAL/busy_timeout/autocommit）與 `PgDriver`（**asyncpg**）。
   - **佔位符正規化**：SQL 一律用 `?`；`PgDriver` 在送出前把 `?` 轉成 `$1,$2,…`。
   - **取回自增 id**：把 `cur.lastrowid` 抽成 `insert()`；SQLite 用 `lastrowid`，PG 用
     `INSERT … RETURNING id`。
2. **schema 方言化**（`_schema.py`）：
   - `INTEGER PRIMARY KEY AUTOINCREMENT` → SQLite 原樣 / PG `BIGINT GENERATED BY DEFAULT AS IDENTITY`。
   - JSON 欄位：SQLite `TEXT` / PG `JSONB`（store 內部仍以 `json.dumps/loads` 介接，行為一致）。
   - `ON CONFLICT(col) DO UPDATE`：兩邊語法相容，保留。`INSERT OR IGNORE` → `ON CONFLICT DO NOTHING`。
   - 時間欄位維持 `REAL`（epoch 秒），避免 timezone 差異。
3. **migration 改寫**：`PRAGMA table_info` 只在 SQLite 用；PG 走 `information_schema.columns`。
   兩邊都收斂到一個 `ensure_column(table, col, type)`。
4. `LLMOpsStore.__init__` 依 `db_path`（檔案/`:memory:`）或 `db_url`（`postgres://`）選 driver；
   `init()` 套 schema + migration。**所有現有方法不改邏輯,只改成走 driver。**

### 測試
- **同一套 store 測試跑兩種後端**：把 `packages/llmops-store/tests/test_store.py` 的 `store`
  fixture 參數化（`params=["sqlite","postgres"]`）。Postgres 用 **testcontainers-python** 起一個
  臨時 PG（無則 `pytest.skip`，並在 CI 用 service 容器跑）。→ 確保 PG 實作與 SQLite **行為等價**。
- 既有 backend(330+)/router 測試**全部仍跑 SQLite**,須 0 退化。
- driver 單元測試：`?`→`$n` 轉換、`insert() RETURNING`、`ensure_column` 冪等。

### 風險 / 驗收
- 風險：方言遺漏（某條 SQL 在 PG 失敗）→ 用「雙後端跑同一套測試」逼出來。
- 驗收：`LLMOPS_DB_URL=postgres://…` 啟動,所有功能（金鑰/稽核/版本/成本/eval/perf）正常;不設則 SQLite 不變。
- 新依賴：`asyncpg`（backend + router requirements）；`testcontainers`（dev 測試用）。

## 2b. Overlay 入庫 + desired 持久化 / 重放

**目標**：多副本看到同一份權威設定;重啟 / 接管能把「該跑的」拉回來。

### 做法
1. **Overlay 權威來源改 DB**：
   - 新 `config_overlay` 表（單列 JSONB，或沿用既有 `config_versions` 的最新一筆當「目前」）。
   - [overlay.py](../apps/backend/app/services/overlay.py) 的 `load_overlay`/`save_overlay` 介面**不變**,
     底層在「DB 模式」改讀寫 DB（交易內 read-modify-write,消除檔案 race）；SQLite 單機模式仍可用檔案。
   - `save_overlay` 與既有「設定版本快照」合流（本來就把整份 overlay 存進 `config_versions`）。
2. **desired 狀態入庫**：新 `instance_desired(key, desired, updated_at)` 表；`start/stop/sleep/wake`
   寫入 desired;reconciler 以 DB 的 desired 為準。
3. **開機重放**：lifespan 在 `adopt_running` 後,讀 `instance_desired`,把 `desired=RUNNING` 但目前
   STOPPED 的重新 `start()`（冪等,沿用 Phase 1 的可重入啟停）。→ 補上「desired 重放」(Phase 1 併過來的項目)。

### 測試
- overlay round-trip（DB 模式）：寫入 → 另一個 store handle 讀到同一份。
- 版本快照仍正確（既有 config-versioning 測試在 DB 模式下跑）。
- desired 重放：seed `instance_desired=RUNNING` → 開機流程呼叫 `start()`（用 FakeProc,斷言被叫到）。
- 並發寫 overlay：兩個 handle 同時 `save_overlay`,交易序列化,不互蓋（PG 後端）。

### 風險 / 驗收
- 風險：overlay 從「檔案」變「DB」可能漏改某個直接讀檔的地方（embedding launcher 會 dump effective
  config）→ 全文搜 `dynamic_models.json` / `load_overlay` 確認都走介面。
- 驗收：DB 模式下加/改模型,第二個副本 reload 後看得到;重啟後 desired 模型自動拉回。

## 2c. Leader election（選主）

**目標**：多副本只有 **leader** 跑背景單例迴圈,其餘待命;leader 掛則自動接手。

### 做法
1. **租約機制**（PG）：`leader_lease(name PRIMARY KEY, holder, expires_at)` 一列;
   `try_acquire(holder, ttl)` = `INSERT … ON CONFLICT(name) DO UPDATE SET holder=…,expires_at=…
   WHERE leader_lease.expires_at < now()` → 回傳是否搶到。或用 PG `pg_try_advisory_lock`（更簡,但
   連線中斷才釋放,租約表較可控）。**選租約表**（可觀測、TTL 可調、SQLite 也能模擬給單機測試）。
2. **心跳**：leader 每 `ttl/3` 續租;失敗（被搶走/DB 不可達）即視為失去 leadership。
3. **迴圈閘門**：[main.py](../apps/backend/app/main.py) 把那 6 個 `create_task` 包進一個
   `LeaderLoop`:`is_leader` 才啟動 reconcile/autoscaler/prune/写 Prometheus targets;失去 → 取消這些
   task；follower 仍服務唯讀 API + 路由轉發。
4. **單機退化**：未接 PG（SQLite 模式）→ 永遠是 leader（單副本,無需選主）,行為等同今天。

### 測試
- 兩個 store handle 搶同一租約 → 只有一個拿到;持有者續租成功、非持有者搶不到。
- 過期接管：把 `expires_at` 設過去 → 另一個 handle `try_acquire` 成功。
- `LeaderLoop`:`is_leader=False` 時背景迴圈不啟動;翻成 True 後啟動;翻回 False 後取消（用假 loop 計數）。
- 單機模式:無 PG → `is_leader` 恆真。

### 風險 / 驗收
- 風險：**腦裂**（兩個都自認 leader）—— 用「DB 交易 + `WHERE expires_at < now()`」保證原子搶租;TTL
  設保守（預設 15s,心跳 5s）。reconcile 動作本身已盡量冪等,短暫重疊不致災難。
- 驗收：起兩個 backend 副本,只有一個在 reconcile/autoscale;kill leader,另一個 ≤TTL 內接手。

## 2d. Router 水平擴

**目標**：router 可開多副本放 LB 後面(配合 Phase 1 的 `/health` `/ready`)。

### 做法
- **inflight 接受近似**：每副本各自記自己的 `backend_inflight`(只影響自己這台的選擇);真正的後端
  負載來自 vLLM `/metrics`(本就一致,每副本各自 poll)。放大樣本後偏差可忽略 → **不引入分散式計數**。
- **RR 改無狀態**:`round_robin` 依賴 per-process 游標,多副本會各轉各的;預設策略改建議用
  `least_load`/`p2c`(無狀態),或把 RR 游標也視為「近似可接受」。文件標註 RR 在多副本下為 per-副本。
- **draining 標記**:Phase 1 的 `app.state.draining` 是 per-副本 in-memory;多副本下 backend 的 drain
  需廣播到所有 router 副本,或把 draining 也移到共享 store(小表 `draining(key, expires_at)`,router 選路時查)。**本子階段把 draining 移到共享 store**,讓 drain 對所有 router 副本生效。

### 測試
- draining 經共享 store:一個 router 副本標記 → 另一個副本選路也跳過(用共享 FakeStore/PG)。
- 無狀態策略多副本一致性(p2c/least_load 不依賴 process 狀態)。

### 風險 / 驗收
- 風險:RR 語意在多副本下改變 → 文件講清楚,預設策略不受影響。
- 驗收:2 個 router 副本 + LB,推理正常、drain 對兩副本都生效。

## 2e. 部署 + 遷移 + 文檔

- **compose**:新增可選 `postgres` service（`deploy/docker-compose.ha.yaml` override 或 profile）;
  backend/router 設 `LLMOPS_DB_URL`。單機預設仍不含 PG。
- **資料遷移**:一次性腳本 `scripts/migrate_sqlite_to_pg.py`：讀現有 SQLite,逐表搬進 PG（含
  overlay 檔 → DB）。冪等、可重跑。
- **設定**:
  | env | 說明 |
  |---|---|
  | `LLMOPS_DB_URL` | `postgres://…` 啟用 PG;空 → SQLite(預設) |
  | `LLMOPS_SESSION_SECRET` | 多副本必設(共享 SSO session) |
  | `LLMOPS_LEADER_LEASE_TTL` | 租約秒數(預設 15;心跳 1/3) |
  | `LLMOPS_INSTANCE_ID` | 此副本識別(預設 hostname,當 lease holder) |
- **文檔**:`deployment` 補「HA 部署」一節(起 PG、跑遷移、開 N 副本、放 LB);roadmap 標 Phase 2 完成;
  `.env.example` 補上述變數;features 補「控制平面 HA」。

## 3. 建議執行順序 + 驗收里程碑

| 子階段 | 產出 | 獨立價值 | 風險 |
|---|---|---|---|
| **2a** store 抽象化 + PG | 可切 `LLMOPS_DB_URL`,功能等價 | 高(備份/查詢/共用即得) | 中(方言) |
| **2b** overlay 入庫 + desired 重放 | 多副本同設定;重啟自動拉回 | 中高 | 中 |
| **2c** leader election | 只有 leader 跑迴圈,自動接管 | 高(真 HA) | 中(腦裂) |
| **2d** router 水平擴 | router 多副本 | 中 | 低 |
| **2e** 部署/遷移/文檔 | 一鍵 HA 部署 + 資料搬遷 | 收尾 | 低 |

**驗收(整體)**:`LLMOPS_DB_URL` 指向 PG、起 2 個 backend + 2 個 router 放 LB；kill 任一 backend,
控制平面 ≤TTL 內由另一副本接手、無資料遺失;單機(不設 PG)行為與今天**逐位元一致**。

## 4. 測試策略總覽

- **雙後端等價測試**:store 測試在 SQLite + PG 各跑一遍(PG 用 testcontainers,缺則 skip;CI 用 PG
  service)。這是 2a 的安全網。
- **既有全套 0 退化**:backend / store / router 既有測試持續綠(SQLite 路徑)。
- **HA 行為單元測試**:租約搶佔 / 過期接管 / LeaderLoop 閘門 / desired 重放 / overlay 並發寫 / draining
  經共享 store —— 全部可用假時鐘 + 兩個 store handle 在單進程內模擬,不需真多進程。
- **整合(手動/選用)**:docker 起 PG + 2 副本,實際 kill leader 驗接管。

## 5. 新依賴

- `asyncpg`(backend + router runtime,僅 PG 模式需要;SQLite 模式不載入)。
- `testcontainers`(dev/test;CI 可改用 service 容器)。

> 落地建議:**先只做 2a**（store 抽象化 + Postgres，零行為改變、附帶價值最高、風險可控),跑通「雙後端
> 等價測試」後再決定是否續做 2b→2c。2a 本身就讓你能把資料庫換成 Postgres、備份查詢更乾淨,即使暫不追
> 多副本也值得。
